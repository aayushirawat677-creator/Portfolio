What If? — a decision-exploration tool that never gives advice.

Ask any life question, get both honest futures, explore each timeline card by card — and decide for yourself. I built What If? solo as a testbed for a product thesis I believe in: that the most useful thing an AI can do with a hard decision isn't to answer it, but to help someone see it clearly. Built for Microsoft Agents League 2026 — I framed the problem, designed the workflow, made the product calls, and drove the build with AI coding tools.

01 · The Problem

Every AI decision tool makes the same mistake.

"You should do X." "Option B is better for you." "I recommend…" Every AI tool that touches a decision falls into the same trap — it decides for you.

That's the wrong model. The user knows their own life better than any AI does. What they actually need in a hard decision isn't a verdict — it's clarity: a way to see both paths honestly, without someone else's thumb on the scale. An AI that hands down an answer removes the one thing the user came for, which is the ability to choose.

So I built What If? around a single non-negotiable principle, and let every downstream decision follow from it.

02 · Product Principle

Show what could happen — never what to do.

One rule the entire product is built to protect. If a feature or output ever tips into advice, it's wrong by definition — no matter how helpful it seems.

// the one rule everything serves
output = possibilities, never advice
// the definition of success
success = user_makes_their_own_decision ? true : false

05 · Key Product Decisions

The calls that made it a product, not a demo.

The interesting work wasn't the pipeline — it was the judgment calls about how each stage should behave.

Decision 01 · The AI assigns polarity, not the user

"Skip my workout?" → the system decides yes = negative, no = positive.

If the user framed their own question's polarity, the tool would just mirror their bias back at them. So I made the classifier assign the emotional direction of each path — which keeps both futures honest even when the user is secretly hoping for one answer. For neutral comparisons ("MacBook or Windows?"), polarity is switched off entirely and both sides stay purely factual.

Decision 02 · A hard cap of three questions

After three, generate regardless — the user came for clarity, not a survey.

More context always makes a better output, so the tempting choice is to keep asking. I capped it at three because the product's job is to reduce decision friction, and an interrogation adds friction. The context stage collects for both paths, not just the likely one, so three questions have to be spent well — a constraint that forced the classifier to pre-decide exactly what to ask.

Decision 03 · Specific-or-it-failed

"You'll feel better" is a failure state. "By week 3 your back pain eases" is the bar.

Generic futures are what make AI feel useless, so I defined the output quality bar as specificity: second-person, causally connected, and grounded in the user's actual answers. Every path also has to include real friction — no utopias, no doom — because a future with no obstacles isn't honest, and honesty is the whole product.

06 · Safety & Guardrails

The safety layer runs in code, before any agent sees the input.

A deterministic check is faster and far harder to jailbreak than asking an LLM to police itself. So the first line of defense never reaches the model.

✓
Code-level hard stopsSelf-harm, harm to others, medical decisions, illegal activity and dangerous substances are caught by deterministic pattern-matching in code and routed to real resources — before any agent runs.
✓
Age-gatingAlcohol and tobacco questions gate on age, with a polite redirect for under-age users rather than a generated future.
✓
Never-advise, enforced in promptsThe generator is barred from "should," "recommend," or "best option" — and every scenario must include at least one realistic obstacle.
✓
Why code, not the LLMSafety-critical stops can't depend on a model that can be talked out of them. Judgment and safety live in deterministic code; the LLM only does the synthesis.
07 · Measuring Quality

How I defined "good" — a 7-dimension eval, run on the platform.

To know whether the agents were actually working, I set up an evaluation that scores outputs across seven dimensions — so "quality" was a measured thing, not a vibe. It runs as a separate improvement loop, not in the user's request path.

// 01–02

Structural & Accuracy

Does the output match the required JSON schema, and did the parser get the decision type and polarity right?

// 03–05

Quality, Safety & No-Advice

Are the futures specific and grounded in the user's answers, were dangerous inputs handled, and does it avoid "should / recommend"?

// 06–07

Friction & Diversity

Does every path include a real obstacle, and are the two futures genuinely different from each other?

Each dimension returns a 0–10 score with a pass/fail and a one-line verdict, run across developer test cases spanning binary decisions, option comparisons, and statement assessments. I ran this through Azure AI Foundry's evaluation, alongside its analytics on agent performance — so I could version prompts, compare runs, and tune with evidence instead of guesswork. The principle I bring to any AI product: define "good," measure it, then iterate.

08 · Scope & Risk

What I deliberately didn't build.

A testbed earns its focus by leaving things out. These were conscious cuts, and the one risk I'd watch first if this went further.

—
No accounts or historySingle-session only. Persistent profiles and cross-session memory were out of scope for proving the core interaction.
—
No fine-tuningPrompt design + evals on a base model, not a custom-trained one — the thesis was about interaction and guardrails, not model performance.
!
Biggest risk: path diversityThe real failure mode isn't safety — it's two futures that quietly say the same thing. If the paths aren't genuinely different, the product has no reason to exist, which is why "diversity" is its own eval dimension.
09 · Reflection

What I'd do differently.

The build confirmed the thesis is worth pursuing — the "no advice, both futures" interaction genuinely feels different from every chatbot that rushes to answer. Two things I'd change next time. First, I'd define the eval before the prompts, not after — I tuned the generator by feel early on, then built the 7-dimension eval, and some of that early work I'd have done faster with the scorecard in front of me. Second, I'd pressure-test the 3-question cap — it's the right instinct, but I set it by judgment, not data; the honest next step is to watch where users actually have enough context and let that number earn itself.
